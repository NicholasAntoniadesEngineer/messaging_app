# Secure Messenger

A standalone, end-to-end encrypted (E2EE) web messenger built on Supabase. Messages
and file attachments are encrypted in the browser; the server only ever stores
ciphertext and public keys.

> **Status:** extracted into its own project from a larger app and decoupled to run
> independently. It is under active security hardening — see
> [`SECURITY.md`](SECURITY.md) and [`docs/SECURITY_REVIEW.md`](docs/SECURITY_REVIEW.md)
> (generated by the audit) for the current posture and the remediation roadmap.

---

## What it does

- 1:1 conversations with real-time delivery (Supabase Realtime).
- End-to-end encryption: per-user X25519 identity keys, ECDH-derived per-conversation
  session keys, per-message keys (HKDF-SHA256), message/file encryption with
  XSalsa20-Poly1305 (TweetNaCl `secretbox`).
- Encrypted file attachments (Supabase Storage, auto-expire after 24h).
- Password- and recovery-key-based key backup for multi-device access.
- Contacts/friends list and user blocking.
- In-app new-message notifications.

## Architecture

```
auth/            Sign-in / sign-up / password reset / recovery-key UI
messaging/       Messenger UI, controller, and services (send/receive, attachments, pairing)
encryption/      E2EE engine: key management, derivation, primitives, backups, facade
database/        Supabase client config, generic DB service, schema, edge function
shared/          Auth service/guard, header, logging, validators, small utilities, styles
```

- **No backend of its own** — everything runs client-side against Supabase
  (Postgres + Auth + Realtime + Storage + one Edge Function).
- **Crypto primitives**: TweetNaCl (`nacl`/`nacl-util`) + the Web Crypto API
  (HKDF, PBKDF2, AES-GCM for backups).

## Prerequisites

- A **dedicated** Supabase project (do not share one with another app).
- Node 20+ (for the Vite dev server / build).

## Setup

1. **Create a Supabase project** at <https://supabase.com>.
2. **Configure the client**: edit [`database/config/supabaseConfig.js`](database/config/supabaseConfig.js)
   and set `PROJECT_URL` and `PUBLISHABLE_API_KEY` (Settings → API). The
   publishable/anon key is meant to be public; row-level security protects the data.
3. **Create the Storage bucket** (must exist *before* running the SQL): a **private**
   bucket named `message-attachments` with a 1&nbsp;MB file-size limit
   (see [`database/setup/supabase-storage-setup.md`](database/setup/supabase-storage-setup.md)).
4. **Apply the schema**: run [`database/setup/messaging-schema.sql`](database/setup/messaging-schema.sql)
   in the Supabase SQL editor. It creates all messaging + encryption tables with
   row-level security, the realtime publication, and storage policies.
5. **Deploy the edge function**: `user-lookup`
   (in [`database/supabaseEdgeFunctions/`](database/supabaseEdgeFunctions/)). It resolves
   email ↔ user-id (there is no `profiles` table; identity is `auth.users`).
6. **Configure Auth**: in the dashboard, set the Site URL and add the deployed app URL
   plus `…/auth/views/auth.html` to the redirect allow-list (sign-up confirmation and
   password-reset links depend on this). Configure SMTP / email confirmation as desired.

## Run

```bash
npm install
npm run dev      # opens the messenger; you'll be redirected to sign in
npm run build    # builds CSS (Tailwind) + bundles with Vite into dist/
```

Deployment is via GitHub Pages (`.github/workflows/deploy.yml`) — it runs `npm run build`
and publishes. A push to `main` deploys.

## Security model (summary)

- The server is **zero-knowledge of message plaintext**: it stores only ciphertext,
  nonces, public keys, and password/recovery-encrypted key backups.
- The user's **password is the root secret**: it (via PBKDF2, 600k iterations) unwraps
  the identity secret key and the session backup key. A server-side password reset alone
  does **not** recover E2EE data — the 24-word **recovery key** is the fallback.
- Authenticity/MITM, forward secrecy, at-rest key storage, CDN integrity, and logging
  are areas being actively hardened — see [`SECURITY.md`](SECURITY.md).

## Independence notes (decoupled from the origin project)

This app was extracted from a larger "money tracker" application. The following were
removed/neutralised so it runs standalone:

- Stripe **payments** and **subscription gating** — messaging is ungated.
- The in-chat **budget "Share Data"** feature (sharing months/pots/settings).
- Money-tracker pages and unused shared modules; the schema was reduced to
  messaging/encryption tables only.

Residual internal naming (`moneyTracker*` config file/global names, the
`MoneyTrackerEncryption` IndexedDB name) is cosmetic and tracked for cleanup; it does
not affect functionality or independence.
